Architecture
Two Governance Layers — Not One
Layer 1 — User Permissions: Exists in downstream systems. Access follows credentials. Agents inherit user permissions. (Microsoft Cloud Adoption Framework)
Layer 2 — Agent Governance: New. What Claude can do with those permissions — multi-system queries, action chaining, read/write, prompt injection. Guardrails separate from model logic. (Palo Alto Governance Framework)
↳ Connector walkthrough maps where Layer 1 ends and Layer 2 begins
Timeline
Platform Choice ≠ Speed
Native connectors solve authentication — OAuth, hours. But auth isn't the bottleneck. Server-side governance config is: tool scoping, DLP, cross-system boundaries, audit format. Identical work regardless of platform. Claude Enterprise, Airia, or anything else — the platform affects who manages controls, not whether the work exists.
↳ Scope per-connector lift before committing to timelines
Sequencing
Compliance Defines the Standard
Technology builds the options. Compliance chooses. Going prepared with clear trade-offs produces a better outcome than going early with abstract questions. Everything we log can also be used against us. The CCO sits in front of the examiner — she should define what defensible looks like.
↳ Build the technical picture first, then bring compliance real options
Non-Starter Without Controls
Cowork — Not for Regulated Workloads
Anthropic states: "Do not use Cowork for regulated workloads." Not in audit logs, Compliance API, or data exports. History stored locally outside retention policies. We're a regulated entity — using a product whose vendor says it's not built for us is not a manageable risk. Compensating control: Cortex XDR with Koi integration (when available) provides endpoint-level visibility into AI agent behavior on the device — what Claude modified, when, which files. Does not replace application-level audit from Anthropic, but closes the endpoint blind spot.
↳ Disable until Koi ships in Cortex XDR or Anthropic builds audit coverage. Pursue both.
Non-Starter

But the underlying problem remains until Anthropic builds application-level logging.
↳ Block desktop tools on synced folders. Cortex XDR/Koi provides interim detection when available.

What Airia Provides (Validated March 2026)
MCP Gateway
~1,100 pre-catalogued MCP servers. Admins approve which servers are available org-wide; users/teams configure their subset. Snowflake access restrictable to specific schemas/tables. Containerized MCP server hosting — spun up at runtime, destroyed on completion.
AI Gateway
Full request/response logging across all LLM traffic. Guardrails for prompt injection, sensitive data exposure, policy violations. Per-user and per-gateway dollar-value budgets (not just token limits) with automatic routing to cheaper models at threshold. Failover and load balancing across providers.
Surfaces
Claude Code and Cowork: full control via gateway routing. Web app: browser extension monitoring (Airia acknowledged this is "heavy-handed," not true security). Native desktop app: cannot currently be intercepted inline — compliance API approach on short-term roadmap (see below).
Multi-Model
Not Claude-exclusive. Supports OpenAI, Gemini, self-hosted models on AWS/Azure. Translation layer handles provider switching. Relevant if Westfield diversifies models by use case.
Standalone
MCP Gateway works without Claude Enterprise. Full org-wide policy enforcement through Claude requires Enterprise licensing. Other investment firms use hybrid: Enterprise for most users, Airia layered on for governed data access.
Must Configure
What Westfield Still Has to Define in Airia
Airia provides the platform. Someone still has to define the rules. This is the Layer 2 agent governance work — it lives in Airia's console instead of being built from scratch, but the decisions are the same.
Agent-to-system access policies — which agents can access which systems. Snowflake read-only vs. Slack read-write vs. FactSet read-only. Per-role, per-team, per-workflow.
DLP rules and data classification — what data can leave each system and reach the model. PII, client account numbers, sensitive fields. Define once in Airia, enforce everywhere.
Cross-system boundary policies — rules for when a single prompt touches multiple systems. Client data silo enforcement at the query level.
Approval workflows — which agent actions require human confirmation before execution. Write operations, cross-system aggregation, high-risk data access.
Risk classification — tag agents, models, and data sources with Westfield's risk definitions. Align with internal compliance policies.
Routing rules — route requests based on data sensitivity, team, and compliance requirements. Set cost thresholds and latency targets per workload.

Architecture Fit

Airia IS the agent governance layer. It sits between Claude and every downstream system, enforcing DLP, cross-system boundaries, tool scoping, and audit logging. This is what we've been describing — commercialized.
